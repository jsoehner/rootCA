## Enterprise Auto-Enrollment Gotchas (Lessons Learned)

During the execution of the `Test-Enterprise.ps1` automated validation suite, we encountered several critical cryptographic boundary issues related to running an **ECDSA (ECC P-384)** Enterprise CA against default Active Directory V1 templates. 

These gotchas cost substantial time to debug and should serve as a reference for any future PKI engineers deploying ECC AD CS topologies:

1. **The RSA-to-ECDSA CryptoAPI Chain Validation Bug (`0x80070057`)**
   - **The Error:** When a client uses `Get-Certificate` or `certreq -enroll` with a default V1 template (e.g., `DomainController` or `Machine`), the CA successfully issues the certificate. However, the client immediately throws `ERROR_INVALID_PARAMETER (0x80070057)` and refuses to install it.
   - **The Cause:** V1 AD CS Templates hardcode the client to generate its private key inside a *Legacy Cryptographic Service Provider (CSP)* (e.g., "Microsoft RSA SChannel Cryptographic Provider"). When the client receives the newly issued certificate, the local CryptoAPI intercepts it and attempts to validate the CA's signature to build the trust chain. Because the CA signed the certificate using ECDSA, the Legacy CSP completely chokes—it does not support ECC signature algorithms—causing the installation to fail.

2. **The `certreq` Template Parsing Crash (`NTE_PROV_TYPE_NOT_DEF`)**
   - **The Error:** Attempting to force the client to use a modern CNG provider (which supports ECC signatures) by using an INF file with `ProviderName = "Microsoft Software Key Storage Provider"` and `CertificateTemplate = DomainController` causes `certreq -new` to crash with `NTE_PROV_TYPE_NOT_DEF`.
   - **The Cause:** `certreq.exe` queries Active Directory for the template attributes during the CSR generation phase. It sees the template demands a Legacy CSP (ProviderType 1) and attempts to illegally merge this requirement with your INF's CNG Provider (ProviderType 0). The collision crashes the generator.

3. **The Solution: The Decoupled Enrollment Pattern**
   To securely auto-enroll certificates from an ECDSA Enterprise CA without upgrading all V1 templates to V3/V4 CNG templates, we developed the **Decoupled Enrollment** workflow:
   - **Step 1 (Generate):** Use an INF file to explicitly generate an RSA key inside a CNG KSP (`ProviderName = "Microsoft Software Key Storage Provider"`). **Crucially, do not include the `[RequestAttributes]` block in the INF.** This blinds `certreq -new` to AD, bypassing the parsing crash.
   - **Step 2 (Submit):** Execute `certreq.exe -machine -submit -attrib "CertificateTemplate:DomainController" req.csr`. The `-machine` flag ensures the CA queries the computer's AD object for SANs (like DNS Name). The `-attrib` flag injects the template requirement only at the CA level, isolating the client from the template's CSP restrictions.
   - **Step 3 (Accept):** Execute `certreq.exe -machine -accept`. The CNG KSP easily validates the ECDSA signature from the CA and installs the payload perfectly.

4. **Renewal Fallback Bug (`0x80070057` redux)**
   - **The Error:** Natively running `certreq -enroll renew` on the successfully installed certificate triggers the `0x80070057` bug again.
   - **The Cause:** The native renewal engine queries the AD Template again, overrides the existing certificate's CNG context, and attempts to generate the renewal key inside a Legacy CSP.
   - **The Solution:** Renewals must be executed manually via a `RequestType = CMC` INF file that specifies `RenewalCert = "Thumbprint"`. The CMC format allows `certreq` to inherit the modern CNG cryptography context directly from the existing certificate.
