## 1. Overview

Phase 2 defines the certificate profiles and constraints that EJBCA will use to issue the root CA certificate and subordinate intermediate CA certificates. All profiles are created in EJBCA and tested (including constraint validation and extension rendering) before Phase 3 pilot testing.

**Deliverables:**
- Root CA certificate profile (ECC P-384 + SHA-384, CA-only usages, 10-year validity)
- Subordinate CA certificate profile (ECC P-384 or RSA 4096 per fallback decision, pathLen=0, 5-year validity)
- Extension policy document (AIA/CRL/OCSP strategy, publication endpoints, refresh cadence)
- Pilot test profiles (matching production but with 90-day validity for rapid iteration)
- Formal profile validation checklist (all extensions render correctly, constraints enforced)

## 2. Root CA Certificate Profile

### 2.1 Profile Naming and Scope

| Parameter | Value |
|-----------|-------|
| **EJBCA Profile Name** | `RootCAProd-ECC384-SHA384` (production) or fallback variants below |
| **Certificate Subject** | `CN=JSIGROUP Root CA,O=JSIGROUP,C=CA` |
| **Validity** | 10 years from ceremony date (e.g., 2026-04-19 to 2036-04-19) |
| **Signature Algorithm** | ECDSA with SHA-384 (OID 1.2.840.10045.4.3.3) |
| **Key Size** | ECC P-384 (NIST FIPS 186-4, 384-bit curve) |
| **Self-Signed** | Yes (root certificate must be self-signed) |
| **Serial Number** | Generated by EJBCA (recommend: 8 bytes, random) |

### 2.2 Certificate Extensions

#### Critical Extensions

| Extension | OID | Value | Encoding | Rationale |
|-----------|-----|-------|----------|-----------|
| **Basic Constraints** | 2.5.29.19 | `ca:TRUE, pathLen:NONE` | CRITICAL | Identifies as root CA; no path length limit |
| **Key Usage** | 2.5.29.15 | `digitalSignature, keyCertSign, cRLSign` | CRITICAL | CA-only operations; no TLS/code signing |

#### Non-Critical Extensions

| Extension | OID | Value | Notes |
|-----------|-----|-------|-------|
| **Subject Key Identifier** | 2.5.29.14 | SHA1(root public key) | Enables chain building (AKID matching) |
| **Authority Key Identifier** | 2.5.29.35 | Same as SKID (self-signed) | Self-referential; not enforced for root but recommended |

  - **Validity:** `3650` days (10 years; EJBCA calculates NotAfter from ceremony date)

4. **Extensions Tab:**

## 3. Subordinate CA Certificate Profile (ECC Primary)

### 3.1 Profile Configuration

| Parameter | Value (ECC P-384) | Fallback Value (RSA 4096) |
|-----------|-------------------|--------------------------|
| **Profile Name** | `SubordCAProd-ECC384-SHA384` | `SubordCAProd-RSA4096-SHA256` |
| **Subject DN** | `CN=JSIGROUP Intermediate CA - AD CS,O=JSIGROUP,OU=Certificate Authority,C=CA` | Same |
| **Validity** | 5 years (1825 days) | Same |
| **Signature Algorithm** | SHA384withECDSA | SHA256withRSA |
| **Key Algorithm** | ECDSA | RSA |
| **Key Size** | 384-bit (NIST P-384) | 4096-bit |
| **Serial Number** | 8 bytes, random | Same |
| **Issued By** | Root CA private key (on HSM) | Same |

### 3.2 Critical Extensions

| Extension | Value | Purpose |
|-----------|-------|---------|
| **Basic Constraints** | `ca:TRUE, pathLen:0` (CRITICAL) | Prohibit further subordinate CAs; only end-entity issuance below this CA |
| **Key Usage** | `keyCertSign, cRLSign` (CRITICAL) | Restrict to CA operations |
| **Extended Key Usage** | *(not present)* | Inherited by end-entities at AD CS layer |

### 3.3 Non-Critical Extensions

| Extension | Value | Notes |
|-----------|-------|-------|
| **Subject Key Identifier** | SHA1(subordinate public key) | Enables chain building for end-entity certs |
| **Authority Key Identifier** | Root CA SKID + Root Serial | Point to root in chain-building queries |
| **Certificate Policies** | Optional: OID 2.5.4.97 (org-specific) | Marks subordinate issuance scope |
| **CRL Distribution Points** | `http://ca.jsiggroup.local/crl/root.crl` | Points to root CRL (issued by root, not subordinate) |
| **Authority Info Access** | `caIssuers: http://ca.jsiggroup.local/root.cer` | Link to root cert for chain building |
| **Subject Alternative Name** | *(not present for CA cert)* | Not applicable to subordinate CA |

## 6. CRL and AIA Strategy

### 6.1 Root CRL Publication

| Aspect | Specification |
|--------|---------------|
| **CRL Issuer** | RootCA (self-issued; signed by root private key) |
| **Distribution Points** | `http://ca.jsiggroup.local/crl/root.crl` (static HTTP, no dynamic endpoints) |
| **Cadence** | Issued annually (minimum); on revocation events (unscheduled) |
| **Validity** | Next CRL must be published ≥7 days before current CRL expiry |
| **Revocation Entries** | None (production root never revoked; at expiry, new root ceremony conducted) |
| **Signature Algorithm** | SHA384withECDSA (matches root signature algorithm) |
| **Distribution Method** | Offline signing (Phase 4/6 key ceremony): root signs CRL on HSM; transferred to distribution point via secure media (USB key, airgapped copy) |

### 6.2 Authority Information Access (AIA)

| AIA Extension | Value | Purpose |
|---------------|-------|---------|
| **caIssuers** | `http://ca.jsiggroup.local/root.cer` | Enables chain building: if client encounters subordinate CA cert without root in chain, it can auto-download root |
| **OCSP** | *(not present)* | Offline root does not run OCSP responder; clients rely on CRL checking for revocation status |

### 6.3 Subordinate CRL Distribution

**Strategy:** Subordinate CA (AD CS) issues its own CRL (containing end-entity revocations). Root CRL (from EJBCA root) contains no entries (root is never revoked until expiry).

| Item | Value |
|------|-------|
| **Root CRL:** | Contains revocation entries for subordinate CA (if ever revoked); issued by root key during ceremonies |
| **Subordinate CRL:** | Issued by AD CS; contains end-entity revocations; available via Windows CA endpoint or custom distribution point |
| **Chain Building:** | Client obtains root cert via AIA caIssuers endpoint (root.cer); verifies CRL from root endpoint (root.crl); verifies subordinate CRL from AD CS endpoint |
